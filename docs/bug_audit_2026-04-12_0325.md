## Fixed bugs

### 1. `scripts/cascade_match.py` — "NA" sentinel leaking into filters (HIGH)
**Lines**: 120-130 (find_match preamble).
**Problem**: Anakin uses literal `"NA"` for missing fields. Code used `ana_sku.get("Brand") or ""` which returns the string `"NA"` (truthy), leading to:
- Brand filter matching on literal `"na"` tokens.
- `ana_sku.get("Unit_Value") not in (None, "", "NA", "nan")` missed `"null"`, `"none"`, and case variants.

**Fix**: Added `clean_str()` and a new `parse_num()` helper, then routed all Anakin field reads through them. Also replaced the manual `float()` block with `parse_num()` so currency prefixes (`₹`, `Rs.`, `100/-`) and Indian lakh format (`1,00,000`) are handled consistently.

### 2. `scripts/cascade_match.py` — `parse_unit("1/2 kg")` returning wrong value (HIGH)
**Lines**: 51-78.
**Problem**: Half-unit notation `"1/2 kg"` was parsed as `(2.0, 'kg')` because the regex greedy-matched `"2 kg"`. Off by 4x vs truth.
**Fix**: Added a fraction pre-pass (`^(\d+)/(\d+)\s+(.+)$` → `f"{num/den} {unit}"`). Verified: `"1/2 kg" → (0.5, 'kg')`, `"1/4 ltr" → (0.25, 'l')`.

### 3. `scripts/stage3_match.py` — same `parse_unit("1/2 kg")` bug
**Lines**: 64-82.
**Fix**: Same fraction pre-pass applied.

### 4. `scripts/stage3_match.py` — `parse_num()` missed `Rs.` / `/-` / `Rs` (MEDIUM)
**Lines**: 101-107.
**Problem**: Didn't handle `"Rs. 100"`, `"100/-"`, `"Rs 200"`. Also fine for `"1,00,000"` (passes) — fine.
**Fix**: Replace `Rs.`, `Rs`, then `rstrip("/-")`.

### 5. `scripts/compare_pdp.py` — `parse_num()` same gaps
**Lines**: 28-34.
**Fix**: Same expansion as (4).

### 6. `scripts/compare_pdp_jiomart.py` — `parse_num()` same gaps
**Lines**: 34-40.
**Fix**: Same expansion as (4).

### 7. `scripts/export_pdp_csv.py` — crash on empty SAM PDP file
**Lines**: 124-127.
**Problem**: `csv.DictWriter(f, fieldnames=list(rows[0].keys()))` throws `IndexError` if `rows` is empty (fresh run before any products scraped).
**Fix**: Early return with stderr notice when `rows` is empty. Also cleaned up NA sentinel strings (`Item_Name`, `Brand`, etc.) via new `clean_str()` helper so output CSVs don't leak `"NA NA"` into the `anakin_weight` column.

### 8. `scripts/export_pdp_csv.py` — `parse_num()` same gaps as (4)
**Fix**: Same expansion.

### 9. `scripts/export_review_queue.py` — NA sentinel in name similarity / CSV output
**Lines**: 56-72, 146-156.
**Problem**: `find_top_candidates` used `ana_sku.get("Brand") or ""` — brand `"NA"` would boost any name containing `"na"` substring by +0.1. Also CSV rows had raw `"NA"` strings from Anakin.
**Fix**: Added `clean_str()` helper; routed brand/name/weight reads through it.

## Deferred bugs / observations

### D1. `scripts/export_review_queue.py` — O(N*M) SequenceMatcher runtime
**Lines**: 56-84.
Running `find_top_candidates` for 1000+ unmatched SKUs × 800 SAM products × 2 SequenceMatcher calls is slow (minutes wall-clock). Could be bucketed by brand or first-word index. Not a correctness bug — deferred.

### D2. `backend/app/models/product.py` — no default for `price`
**Line 9**: `price: float` is required. If a scraper computes price = 0.0 it still passes Pydantic validation and the 0-priced product is stored. Not a crash — but silent data quality issue. Could add `@validator` to reject `price <= 0`. Deferred (risk of rejecting legitimate free samples / zero-price promos).

### D3. `backend/app/scrapers/base_scraper.py` — unchanged per instructions
Observations (read-only, not fixed):
- `_parse_generic_product()` lines 272-307: price heuristic `if price > 50000: price /= 100` assumes cents encoding but could mis-correct legitimate large prices (luxury goods, bulk packs). Document known failure mode.
- Line 316: brand fallback `brand = name.split()[0]` — fine for branded SKUs, breaks for generics like `"Fresh Tomato 1kg"` → brand `"Fresh"`.
- Line 455 `_process_responses`: Uses `str(rp.get("id") or rp.get("product_id")...)` — evaluates `rp.get("id")` first, but if `id` is the literal int 0 it evaluates falsy and falls through. Low risk in practice.

### D4. `backend/app/services/compare_service.py` — `parse_price` narrow
**Line 166-172**: Only handles `₹` and comma; not `Rs.`, `/-`, or lakh format. Deferred because reference Excel files typically use numeric cell types, so string parsing is rare edge path.

### D5. `scripts/compare_sam_vs_anakin.py` — legacy, not fixed
Superseded by stage-based approach per brief. Not audited in depth.

## Recommendations

1. **Centralize helpers**: `clean_str`, `parse_num`, `parse_unit` are duplicated across 5+ scripts with slightly different behavior. Move into a shared `scripts/_helpers.py` module so fixes propagate.
2. **Add unit tests for `parse_unit` / `parse_num`**: The `"1/2 kg"` bug would have been caught immediately. Suggested cases:
   - `"1/2 kg"`, `"1/4 ltr"`, `"500g"`, `"1.5 KG"`, `"3 x 100 ml + 1 free"`, `"2×250ml"`.
   - `"₹ 100"`, `"Rs. 100"`, `"100/-"`, `"1,00,000"`, `"nan"`, `"NA"`, `""`, `None`.
3. **`export_review_queue.py` perf**: Pre-index SAM pool by normalized brand before the per-SKU loop — O(N+M) instead of O(N*M).
4. **Consider `Blinkit_Factor` / `Jiomart_Factor`** fields fetched by `fetch_anakin_*.py` — they exist in the schema but aren't consumed by any comparison script. Either drop or integrate.
5. **Blinkit_Partial / Jiomart_Partial** — same as above. Unused.
